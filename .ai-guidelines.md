# AI Guidelines

These guidelines exist to keep AI-generated changes aligned with the project's **V1 spec** and **layered architecture**.
Keep changes minimal, deterministic, and consistent with existing decisions.

## Architecture (Non-negotiable)

The project follows strict layering:

- **UI** → **App (Orchestrator)** → **Engine** → **Domain**
- **Engine** → **Persistence**
- **Engine** → **WinApi**
- **Persistence** → **Domain**
- **Domain** depends on nothing
- **WinApi** depends on nothing

Hard rules:
- No circular references.
- **UI must only reference App** (never Engine/Persistence/WinApi).
- **All OS interaction stays in WinApi** (no `HWND`, `IntPtr`, P/Invoke, Win32 structs outside WinApi).
- **Domain stays pure** (no file I/O, no Win32, no UI, no JSON libs required by Domain).

## V1 Behavioral Commitments (Do not change)

- Platform: **Windows-only**.
- Capture: **create-only**.
- Recapture: **update-only**, **overwrite** semantics.
- Capture/Recapture: store explicit window state (`Normal`, `Minimized`, `Maximized`) per window.
- Restore:
  - **Global close** of visible top-level windows (sequential `WM_CLOSE`),
  - **Abort on refusal/timeout**,
  - Launch executables for entries,
  - Best-effort positioning of **up to N windows** per executable,
  - Replay explicit window state (`Normal`, `Minimized`, `Maximized`),
  - Clamp off-screen windows; **minimize if fully off-screen**,
  - Replay **z-order** after positioning.

If a requested change contradicts these commitments, propose an ADR/backlog item instead of changing V1 behavior.

## Implementation Style

- Prefer small, cohesive types and explicit interfaces at layer boundaries.
- Keep WinApi wrappers thin; push logic into Engine where testable.
- Favor deterministic behavior over clever heuristics.
- Avoid introducing new abstractions unless a second use-case exists.

## Testing Philosophy (V1)

- Prioritize rapid iteration and manual OS validation.
- Unit tests where they add leverage:
  - Domain logic
  - Engine decision logic (mock WinApi/Persistence interfaces)
- Avoid brittle UI automation and direct WinApi “unit tests” in V1.
- Integration tests (optional) may be manual-run smoke tests.

## Documentation

- Do not duplicate content across files.
- Update docs only when a change affects public behavior, architecture, or milestones.
- Prefer adding an ADR for important decisions.
